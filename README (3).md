# 🛍️ Customer Shopping Behavior Analysis

An end-to-end data analytics project that turns a raw customer shopping dataset into actionable business insight — using **Python**, **PostgreSQL**, and **Power BI**.

The project covers the full analytics workflow: loading and cleaning data, engineering features, running SQL business queries, and presenting results in an interactive dashboard.

---

## 📌 Overview

Retailers collect large amounts of transaction data but often struggle to turn it into clear decisions. This project analyzes a customer shopping dataset to answer real business questions:

- Who are the customers, and what do they spend?
- How do discounts and subscriptions affect behavior?
- Which products and customer segments drive the most revenue?

The result is a clean, queryable database and an interactive Power BI dashboard that a business user can explore.

---

## 📊 Dataset

- **Records:** ~3,900 customers
- **Original columns:** 18
- **Format:** CSV (`customer_shopping_behavior.csv`)

**Key fields used:**

| Field | Description |
|-------|-------------|
| `gender`, `age` → `age_group` | Customer demographics |
| `category`, `item_purchased` | What was bought |
| `purchase_amount` | Spend per transaction (USD) |
| `review_rating` | Product rating |
| `subscription_status` | Subscriber vs non-subscriber |
| `discount_applied` | Whether a discount was used |
| `shipping_type` | Shipping method chosen |
| `previous_purchases` | Repeat-buyer history |
| `frequency_of_purchases` → days | How often they buy |

**Engineered features:**
- `age_group` — customers binned into Young Adult, Adult, Middle-aged, and Senior using quartiles.
- `purchase_frequency_days` — text frequency (e.g. "Weekly") mapped to a number of days.

## 🔄 Steps

1. **Load** the raw CSV into a pandas DataFrame in Jupyter Notebook.
2. **Explore (EDA)** — inspect structure, data types, and missing values with `.info()` and `.isnull().sum()`.
3. **Clean** — fill missing review ratings using the category median, standardize column names to lowercase with underscores, and drop a redundant column.
4. **Feature engineer** — create `age_group` and `purchase_frequency_days`.
5. **Load to database** — push the cleaned DataFrame into a PostgreSQL table using SQLAlchemy.
6. **Analyze in SQL** — answer 10 business questions (see below).
7. **Visualize** — connect Power BI to PostgreSQL, build DAX measures, and design an interactive dashboard.

## 🔍 Results — Business Questions Answered

The SQL analysis (`SQL_QUERIES_CUSTOMER_BEHAVIOR_PROJECT.sql`) answers:

1. Total revenue generated by male vs. female customers
2. Customers who used a discount but still spent above the average
3. Top 5 products by average review rating
4. Average purchase amount: Standard vs. Express shipping
5. Whether subscribers spend more than non-subscribers
6. Top 5 products with the highest discount usage rate
7. Customer segmentation into New, Returning, and Loyal
8. Top 3 most purchased products within each category (window functions)
9. Whether repeat buyers are also likely to subscribe
10. Revenue contribution of each age group

## ▶️ How to Run

### Prerequisites
- Python 3.10+
- PostgreSQL
- Power BI Desktop
- Python packages: `pandas`, `sqlalchemy`, `psycopg2-binary`

### Steps

```bash
# 1. Clone the repository
git clone https://github.com/<your-username>/customer-shopping-behavior-analysis.git
cd customer-shopping-behavior-analysis

# 2. Install dependencies
pip install pandas sqlalchemy psycopg2-binary
```

3. **Create a PostgreSQL database** named `customer_behavior`.
4. **Open the notebook** (`CUSTOMER_BEHAVIOR_ANALYSIS.ipynb`) and run all cells to clean the data and load it into PostgreSQL.  
   *Update the connection string with your own username, password, host, port, and database name.*
5. **Run the SQL queries** in `SQL_QUERIES_CUSTOMER_BEHAVIOR_PROJECT.sql` against the database.
6. **Open `REPORT_POWERBI.pbix`** in Power BI Desktop and refresh the data source to connect to your database.

## 💡 Lessons Learned

- **Clean data first** — consistent column names and early null handling prevented errors downstream.
- **Small details matter** — a single trailing space in a database name caused repeated connection failures.
- **Tools are stronger together** — Python, SQL, and Power BI each did one job well; the value came from chaining them into one pipeline.
- **Start from the question** — defining business questions up front kept the analysis focused on insight, not just charts.
